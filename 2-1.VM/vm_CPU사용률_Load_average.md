# CPU 사용량, 코어 수, 및 Load Average 관계

## 1. 핵심 개념 요약
| 항목 | 정의 | 특징 |
| :--- | :--- | :--- |
| **CPU 사용량 (%)** | 프로세스가 CPU 자원을 점유한 비율 | 현재 작업의 **활동성** 지표 |
| **코어 수 (vCPU)** | VM에 할당된 물리/가상 연산 장치 수 | 시스템의 **병렬 처리 능력** 결정 |
| **Load Average** | 실행 중이거나 대기 중인 프로세스의 평균 수 | 시스템의 **전반적인 부하(체증)** 지표 |

### 💡 Load Average 해석 기준
> **Load Average ÷ 코어 수 = 부하율**
- **1.0 (100%)**: 시스템 자원을 남김없이 최적으로 사용 중
- **1.0 미만**: 자원에 여유가 있는 상태
- **1.0 초과**: 자원 경합 발생 (프로세스가 줄을 서서 대기 중)

## 2. CPU 사용량 vs Load Average 상관관계

### 상황 A: CPU 사용량 ↑ / Load Average ↓ (정상/양호)
*   **원인**: 시스템 리소스가 충분하여 작업이 대기 없이 즉시 처리됨.
*   **주요 케이스**:
    1.  **멀티코어 활용**: 코어 수가 많아 여러 프로세스가 병렬로 분산 처리됨.
    2.  **CPU Bound 작업**: 순수 연산(계산) 위주의 작업이며 I/O 병목이 없음.
    3.  **대기열 없음**: 실행 중인 프로세스는 바쁘지만, 실행을 기다리는 프로세스는 없음.

### 상황 B: CPU 사용량 ↑ / Load Average ↑ (과부하/임계)
*   **원인**: CPU 연산 자원 자체가 부족하여 처리 속도보다 요청이 더 많이 들어옴.
*   **주요 케이스**:
    1.  **연산 집중형 프로세스 누적**: 복잡한 계산이나 데이터 처리를 수행하는 프로세스가 코어 수보다 많음.
    2.  **무한 루프/좀비 프로세스**: 비정상적인 프로세스가 CPU를 점유하며 대기열을 생성.
    3.  **리소스 부족**: 현재 스펙으로 감당하기 어려운 트래픽이나 작업량이 유입됨.

### 상황 C: CPU 사용량 ↓ / Load Average ↑ (위험/병목)
*   **원인**: CPU 연산 자체보다는 외부 요인으로 인해 프로세스가 멈춰(Wait) 있음.
*   **주요 케이스**:
    1.  **I/O Wait (Disk/Network)**: 디스크 읽기/쓰기 속도가 느려 프로세스가 `Uninterruptible Sleep (D state)` 상태로 대기.
    2.  **Disk I/O 병목**: CPU는 놀고 있지만, 디스크 처리량이 한계에 도달하여 프로세스 대기열이 길어짐.
    2.  **자원 잠금(Lock)**: 특정 자원을 점유하기 위해 프로세스들이 대기 상태로 누적됨.

## 3. 부하 단계별 상태 및 조치 가이드

| 부하 단계 | Load Average 수준 | 내부 상태 | 예상 결과 | 권장 조치 |
| :--- | :--- | :--- | :--- | :--- |
| **1. 포화** | **코어 수 × 1.0** | 모든 코어가 풀가동 중 | 최적의 처리량 유지 | 모니터링 유지 |
| **2. 경합** | **코어 수 × 2.0** | 런큐(Run-queue) 대기 발생 | 응답 지연(Latency) 체감 | 프로세스 최적화 검토 |
| **3. 임계** | **코어 수 × 3.0** | 컨텍스트 스위칭 오버헤드 급증 | 서비스 타임아웃 빈번 | **Scale-up/out 즉시 실행** |
| **4. 고립** | **코어 수 × 4.0↑** | 커널 스케줄러 제어권 상실 | **System Hang (먹통)** | 하드 리셋 / 콘솔 접속 |

## 4. 리눅스 서버 내 확인 방법 (명령어)

### 1) `top` 또는 `htop`
- 시스템 전체의 CPU 사용률, Load Average, 프로세스별 점유율을 실시간으로 확인합니다.
- **Load Average**: 우측 상단 표시 (1분, 5분, 15분 평균)
- **CPU 사용률**: `%Cpu(s)` 항목 확인

### 2) `uptime`
- 현재 시간, 가동 시간, 접속자 수와 함께 **Load Average**만 간결하게 확인 시 사용합니다.
```bash
$ uptime
 15:20:01 up 10 days,  2:10,  1 user,  load average: 0.52, 0.40, 0.35
```

### 3) `lscpu` 또는 `nproc`
- 현재 시스템의 **전체 코어 수**를 확인하여 Load Average 수치와 비교합니다.
```bash
$ nproc
4  # 코어 수가 4개라면 Load Average 4.0이 100% 포화 상태임을 의미
```

### 4) `vmstat`
- 시스템의 프로세스, 메모리, 페이징, I/O, CPU 활동 상태를 요약해서 보여줍니다.
- `r` (실행 대기 큐): Load Average와 직결되는 실행 대기 프로세스 수
- `wa` (I/O Wait): CPU 사용률은 낮으나 Load가 높을 때 디스크 병목 여부 확인
```bash
$ vmstat 1 5  # 1초 간격으로 5번 출력
```

---

## 💡 결론
1.  **CPU 사용량**만 보지 말고, 반드시 **코어 수 대비 Load Average**를 함께 확인해야 합니다.
2.  CPU 사용량이 100%여도 Load Average가 코어 수 수준이라면 시스템은 **매우 효율적으로 일하는 중**입니다.
3.  반면, CPU 사용량이 낮은데 Load Average가 높다면 **디스크 I/O나 네트워크 병목**을 의심해야 합니다.
