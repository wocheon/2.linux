# Korean Sentiment Classifier 작업 정리 (Playbook)

## 0. 목표/스코프
- 목표: 한국어 뉴스 텍스트를 3-class(Positive/Neutral/Negative)로 분류하는 모델 학습 및 서빙 파이프라인을 안정화한다.
- 범위:
  - 데이터 전처리 포맷 표준화
  - 학습 스크립트(Trainer) 안정화 및 로그 품질 개선
  - 추론 API에서의 입력 포맷/토크나이저 일관성 보장
  - 성능 개선(하이퍼파라미터/데이터 품질)

## 1. 입력 포맷 표준화 (Input Format)
### 개념
- 모델이 `[Target] Title`처럼 특수문자 구분자에 의존하면, 토크나이저/모델별로 토큰 분해가 달라져 성능/일관성이 흔들릴 수 있다.
- 따라서 “자연어 문맥”으로 Target 정보를 주입하는 형식으로 통일한다.

### 표준 포맷
- 권장: `{Target} 관련 뉴스: {Title}`
- 예시:
  - `대우건설 관련 뉴스: 대우건설, 실적과 수주...`
  - `반도체 관련 뉴스: 美 30년 넘게 반도체...`

### 장단점
- 장점: 모델/토크나이저가 달라도 문맥 기반으로 해석되며, 특수기호 의존이 줄어든다.
- 단점: 토큰 길이가 약간 늘어나며(max_length가 아주 타이트한 경우) 절단(truncation) 확률이 증가한다.

## 2. 학습 코드(Trainer) 핵심 변경사항
### 개념
- 학습 데이터 전처리 단계에서 입력 포맷을 표준화하고, confidence 기반 필터링/라벨 검증을 적용한다.
- 학습/검증 로깅은 “학습 진행률은 유지하되”, 검증 단계에서 불필요한 출력이 쏟아지지 않게 제어한다.

### 구현 체크리스트
- 데이터 필터:
  - `confidence >= MIN_CONFIDENCE`
  - label 유효성(0/1/2)만 허용
- 텍스트 결합:
  - `combined_text = f"{tgt} 관련 뉴스: {title}"`
- 학습/검증 분리:
  - 별도 valid 파일이 있으면 동일 전처리 적용
  - 없으면 train_test_split(seed 고정)
- 모델/토크나이저 로딩:
  - 기본은 AutoTokenizer/AutoModel 사용
  - 체크포인트 로딩 시, 토크나이저 파일 누락 이슈가 없도록 “베이스 토크나이저” 사용 여부를 옵션화하는 것도 고려

### 장단점
- 장점: 재현성(Seed 고정), 품질 통제(confidence), 입력 일관성 확보
- 단점: confidence를 너무 높이면 데이터가 급격히 줄어 학습이 불안정해질 수 있음(라벨별 최소 샘플 수 필요)

## 3. 로깅/진행률(Progress) 정책
### 개념
- 운영 관점에서 “학습 진행률 바는 필요”하지만, “검증 단계에서 step-by-step 출력은 불필요”할 때가 많다.
- 해결 전략:
  - 학습 진행률은 살리고
  - 검증 tqdm/progress 출력은 억제하고
  - 검증 결과는 에폭 단위로 한 번만 요약 출력

### 구현 예시(정책)
- 학습: tqdm ON
- 검증: tqdm OFF
- 출력: `eval_accuracy`, `eval_f1`, `eval_loss`만 에폭 종료 시 1회 출력

### 장단점
- 장점: 로그 가독성/운영 편의성 증가
- 단점: 디버깅 단계에서는 상세 step 로그가 필요할 수 있어 토글 옵션을 두는 게 좋음

## 4. 성능 개선 루프(Iteration Loop)
### 개념
- 성능이 애매할 때는 “에폭만 늘리기”보다 아래 순서가 효과적이다.
  1) 입력 포맷 정합성(공백/구두점 포함)
  2) 데이터 품질(라벨 일관성/노이즈)
  3) 하이퍼파라미터(LR, epochs, batch size)
  4) 모델/토크나이저 로딩 일관성

### 권장 운영 루틴
- 매 실험(run)마다 아래를 기록:
  - 데이터 버전(생성 파이프라인 git hash / 날짜 / confidence 기준 / 라벨 분포)
  - 학습 설정(config.ini)
  - best checkpoint metric 및 confusion matrix(가능하면)
